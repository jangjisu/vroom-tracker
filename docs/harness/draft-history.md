# Harness Draft History

이 문서는 하네스 구조를 설계해 가는 과정의 초안 기록이다.

최종 명세가 아니라, 어떤 생각으로 시작했고 무엇을 바꾸기로 했는지, 어떤 시도를 보류하거나 폐기했는지를 발표 자료로 재구성할 수 있도록 남긴다.

앞으로 사용자가 "기록해줘"라고 요청하면 이 문서에 새로운 항목을 추가한다.

---

## 2026-05-21: 하네스 설계 논의 시작

### 배경

`vroom-tracker` 프로젝트에서 Java/backend 파일을 변경할 때 사용할 작업 하네스를 만들고자 했다.

처음에는 단순히 테스트 유틸이나 백엔드 보조 파일을 만드는 방향처럼 보였으나, 대화가 진행되면서 목표가 더 명확해졌다.

### 참고한 구조

사용자가 이전에 본 하네스 구조는 frontend 작업 흐름에 가까웠다.

해당 구조는 작업을 한 번에 처리하는 방식이 아니라, 여러 단계로 나누어 진행하는 방식이었다.

각 단계가 끝날 때마다 hook을 실행하여 이전 단계의 결과가 원하는 조건을 만족하는지 확인하고, 조건을 통과하면 다음 단계로 넘어가는 흐름이었다.

### 현재 초안

이번 프로젝트에서도 우선 다음 4단계 흐름을 기준으로 하네스 구조를 잡아가기로 한다.

1. 계획 생성 단계
2. 코드 작성 단계
3. 검증 단계
4. Commit 단계

아직 세부 규칙과 스크립트는 확정하지 않는다.

먼저 이 4단계 구조를 기준으로 조금씩 갖춰나가고, 이후 사용자가 "기록해줘"라고 요청하는 시점마다 변경 이유와 결정 내용을 이 문서에 추가한다.

## 2026-05-28: 백엔드 하네스 초안 구체화

### 단계 구조 변경

초기에는 4단계 흐름을 기준으로 시작했지만, 백엔드 작업에서는 API 스펙 영향도 확인이 독립된 단계로 필요하다고 판단했다.

따라서 하네스 흐름을 다음 5단계로 재정리했다.

1. 계획 생성
2. 스펙 영향도 확인
3. 코드 작성
4. 검증
5. Commit

스펙이 실제로 바뀌지 않는 작업이라도, 변경 없음과 그 이유를 확인하는 흐름이 필요하다고 보았다.

### 규칙 문서와 hook 역할 분리

처음에는 `rules/` 문서가 작업 후 체크리스트 역할까지 함께 가지고 있었다.

하지만 진행하면서 기계적으로 판단 가능한 항목은 shell hook으로 옮기고, `rules/`는 설계 판단과 예외 판단을 위한 참고 문서로 남기는 방향으로 정리했다.

예를 들어 테스트 실행, 포맷 검사, 커버리지 기준, `@Data` 금지처럼 성공/실패를 판단할 수 있는 항목은 hook이 담당한다.

반대로 Service 레이어 경계, 외부 API 응답 VO와 내부 DTO 분리, API 성공 코드 실측 원칙처럼 맥락 판단이 필요한 항목은 규칙 문서에 남긴다.

### 코드 작성 단계 hook 정리

코드 작성 단계에서는 다음 항목을 hook으로 점검하는 방향을 잡았다.

- 변경 파일 범위 확인
- Lombok `@Data` 사용 금지
- Service/Controller에서 프로젝트 객체 직접 `new` 생성 제한
- Checkstyle/Spotless 등 code quality tool 실행
- `@RestController` 반환 타입 규칙 확인
- 주요 production Java 파일에 대응 테스트 파일 존재 여부 확인

중간에 `Map`/`HashMap` 사용 승인 규칙도 검토했지만, 실제 작업에서 중요도가 낮고 과도한 제약이 될 수 있다고 판단하여 제거했다.

Service/Controller 직접 `new` 제한은 전면 금지가 아니라 allowlist 방식으로 정리했다.

JDK 컬렉션, 문자열 빌더, 값 객체, 예외 객체처럼 정적 팩토리로 감싸는 의미가 적은 생성은 `harness/config/service-controller-new-allowlist.txt`에 추가해서 허용한다.

### 품질 도구와 커버리지 기준 도입

Gradle에 코드 품질 도구를 직접 연결하는 방향으로 정리했다.

도입한 기준은 다음과 같다.

- Checkstyle: 미사용 import 등 정적 검사 기반
- Spotless + Palantir Java Format: 자동 포맷 적용
- SonarQube: 환경이 준비된 경우 실행 가능한 구조
- JaCoCo: line coverage 최소 60%

포맷처럼 자동으로 바로 고칠 수 있는 문제는 hook에서 먼저 적용하고, 그 이후에도 남는 Checkstyle/Spotless 문제는 실패로 판단하도록 했다.

JaCoCo 60% 기준을 실제로 통과시키기 위해 공통 응답, 예외 처리, 외부 API 응답 VO 테스트를 보강했다.

### Commit 단계 정리 방향

Commit 단계는 과하게 복잡하게 만들지 않고 최소 기준으로 잡기로 했다.

기준은 다음 정도로 정리했다.

- `main`, `master` 직접 커밋 방지
- staged 파일 존재 여부 확인
- 커밋 메시지 제목 형식 확인

커밋 메시지는 기존 프로젝트 히스토리에 맞춰 다음 형식을 사용하기로 했다.

```text
type: 내용
type(scope): 내용
```

허용 type은 `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `merge`로 정했다.

다만 Commit 단계 hook과 Verify 단계 hook 변경은 이번 원격 반영 대상에서 제외하고 로컬 변경으로 남겨두었다.

### 원격 반영 범위

이번에 `origin/main`에 반영한 커밋은 하네스 전체 초안이 아니라, 빌드 품질 도구와 테스트 보강, 포맷 적용 범위로 제한했다.

반영된 커밋은 다음 3개이다.

```text
build: 코드 품질 및 커버리지 도구 설정
test: 공통 응답 커버리지 보강
style: Java 포맷 적용
```

작업 규칙 정리와 코드 작성 단계 하네스 hook 재구성 커밋은 원격 반영 대상에서 제외했다.
