# vroom-tracker 작업 오케스트레이션

이 문서는 구현 작업에서 어떤 Skill과 역할별 에이전트가 언제 실행되는지 설명한다.
실제 실행 규칙은 `.agents/skills/vroom-workflow/SKILL.md`에 있다.

## 어떻게 실행되는가

백엔드, API, 프론트엔드, 버그 수정과 리팩토링처럼 코드 변경이 포함된 요청은 `vroom-workflow` Skill의 대상이다.

```text
계획 생성
→ 스펙 영향도 확인
→ 역할별 계획 리뷰
→ 사용자 승인
→ 역할별 코드 작성
→ 자동 검증과 Compound 리뷰
→ Commit / Push
```

질문, 방향 논의, 코드 설명과 단순 Git 조회는 이 전체 흐름을 실행하지 않는다.

## 계획 리뷰 역할

| 역할 | 실행 기준 | 사용 기능 |
|---|---|---|
| 개발팀장 | 코드 변경이 있는 모든 작업 | `/gstack-plan-eng-review` |
| CEO | 신규 기능, 제품 가치·범위·우선순위 변경 | `/gstack-plan-ceo-review` |
| DevEx | API, 라이브러리, 하네스, 개발자 인터페이스 변경 | `/gstack-plan-devex-review` |
| CSO | 인증, 키, 외부 입력, 공개 API, 개인정보, 의존성, 배포 변경 | `/gstack-cso` |
| Design | 화면 구조 또는 사용자 흐름 변경 | `/gstack-plan-design-review` |

모든 역할을 매번 실행하지 않는다.
개발팀장 리뷰는 항상 실행하고, 나머지는 작업 영향에 따라 선택한다.

각 리뷰는 계획을 평가하는 데서 끝나지 않는다.
지적을 계획에 반영하고, 수정한 계획을 사용자가 승인해야 코드 작성으로 넘어간다.

## 코드 작성 역할

계획의 구현 단위마다 다음 역할을 순서대로 수행한다.

1. **구현 에이전트**
   - 실패하는 테스트를 먼저 작성한다.
   - 테스트를 통과하는 최소 코드를 구현한다.

2. **스펙 리뷰 에이전트**
   - 승인된 계획과 구현이 일치하는지 확인한다.
   - API, 데이터 계약과 작업 범위 누락을 찾는다.

3. **품질 리뷰 에이전트**
   - 책임 경계, 단순성, 테스트 의미와 프로젝트 규칙을 확인한다.
   - 요청하지 않은 리팩토링과 과도한 추상화를 찾는다.

4. **수정 및 재검증**
   - 구현 에이전트가 리뷰 지적을 수정한다.
   - 해당 구현 단위의 테스트와 코드 훅을 다시 실행한다.

하위 에이전트 기능을 사용할 수 없는 환경에서는 한 에이전트가 역할을 섞지 않고, 같은 순서를 별도 체크포인트로 수행한다.

## 최종 검증

- Shell hook: 기계적으로 판단 가능한 프로젝트 규칙
- Gradle과 JaCoCo: 테스트, 전체 커버리지와 변경 코드 커버리지
- Compound Engineering: 정확성, 테스트, 유지보수성, 프로젝트 규칙 등 다중 관점 코드 리뷰
- Compound 학습 기록: 반복해서 사용할 가치가 있는 해결 방법이 있을 때만 작성

## 결과 확인 위치

현재 작업의 계획과 리뷰 결과는 로컬 실행 디렉터리에서 확인한다.

```text
harness/runs/current/
├── state.env
├── reviews/
│   ├── plan-eng-review.md
│   ├── plan-ceo-review.md
│   ├── plan-devex-review.md
│   ├── cso-review.md
│   ├── design-review.md
│   └── compound-code-review.json
└── commit-message.txt
```

필요하지 않은 리뷰 파일은 만들지 않는다.
`state.env`의 `not-required` 값과 계획에 기록된 이유로 생략 여부를 확인한다.

리뷰 상태를 `completed`로 기록하려면 대응하는 결과 파일 경로도 `state.env`에 기록해야 한다.
따라서 상태값만 임의로 바꾸어 리뷰 단계를 통과할 수 없다.
