# rest-route 작업 오케스트레이션

이 문서는 구현 작업에서 어떤 Skill과 역할별 에이전트가 언제 실행되는지 설명한다.
실제 실행 규칙은 `.agents/skills/vroom-workflow/SKILL.md`에 있다.

## 어떻게 실행되는가

백엔드, API, 프론트엔드, 버그 수정과 리팩토링처럼 코드 변경이 포함된 요청은 `vroom-workflow` Skill의 대상이다.

```text
계획 생성
→ 스펙 영향도 확인
→ 사용자 승인
→ TDD 코드 작성
→ 관련 Markdown 문서 동기화
→ 단일 코드 리뷰
→ 자동 검증
→ Commit / Push
```

질문, 방향 논의, 코드 설명과 단순 Git 조회는 이 전체 흐름을 실행하지 않는다.

## 계획 확정

스펙 영향도, 접근 방법과 trade-off를 반영한 계획을 사용자가 승인하면 코드 작성을 시작한다.
역할별 계획 리뷰는 실행하지 않는다.

## 코드 작성

승인된 계획을 TDD의 Red, Green, Refactor 순서로 구현한다.
구현 단위별 스펙 리뷰, 품질 리뷰와 재리뷰는 실행하지 않는다.
집중 테스트로 개발하고 전체 구현이 끝난 뒤 코드 게이트를 한 번 실행한다.

## 문서 동기화와 단일 리뷰

변경된 계약에 영향을 받는 기준 문서를 현재 코드와 맞춘다.

- `PRODUCT.md`: 제품 가치와 범위
- `DATA.md`: 데이터 저장과 연결
- `API.md`: 내부·외부 API 계약
- `ARCHITECTURE.md`: 레이어와 흐름
- `QUALITY.md`: 테스트와 품질 기준

코드 리뷰는 전체 작업에서 한 번만 수행한다.
한 번의 리뷰에서 코드, 테스트, 프로젝트 규칙과 관련 Markdown 문서의 정합성을 함께 확인한다.
결과 파일에는 `문서 정합성` 또는 `Documentation consistency` 항목을 남긴다.

확실한 지적은 수정하지만 재리뷰하지 않는다.
수정 뒤 관련 테스트와 전체 검증을 실행한다.

## 최종 검증

- Shell hook: 기계적으로 판단 가능한 프로젝트 규칙
- Gradle과 JaCoCo: 테스트, 전체 커버리지와 변경 코드 커버리지
- 단일 코드 리뷰 결과와 문서 정합성 항목

## 결과 확인 위치

현재 작업의 계획과 리뷰 결과는 로컬 실행 디렉터리에서 확인한다.

```text
harness/runs/current/
├── state.env
├── reviews/
│   └── code-review.md
└── commit-message.txt
```

리뷰 상태를 `completed`로 기록하려면 대응하는 결과 파일 경로도 `state.env`에 기록해야 한다.
결과 파일에는 코드와 관련 Markdown 문서의 정합성 확인이 있어야 한다.
`CODE_REVIEW_COUNT`는 정확히 `1`이어야 한다.
